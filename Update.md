# Shorts100 업데이트 / 개선 사항 (12인 클로즈드 테스트 기준)

> 작성일: 2026-05-29 | 테스트 단계: 클로즈드 12인

---

## 🔴 높은 우선순위

### [완료] 1. 구글 로그인 불안정
- WebView 환경에서 Google 로그인 에러 발생 시 사용자 친화적인 메시지가 없던 문제 수정.
- **조치 완료**: catch 블록에서 현지화된 메시지("구글 로그인에 실패했습니다. 잠시 후 다시 시도해주세요.")를 세팅하도록 수정.

### [완료] 2. 다운로드 포인트 시스템 UX 불투명
- 게스트 5회 소진 후 제한 안내 및 크레딧 충전 안내가 부족하던 문제 개선.
- **조치 완료**: 다운로드 페이지에 비회원/회원 맞춤형 "다운로드 제한 및 크레딧 충전 안내" 카드 UI를 추가하여 포인트/크레딧 설명 제공.

### [완료] 3. Rising 페이지 제목/설명 불일치
- 실제 기능인 조회수 급등 영상에 맞지 않던 카테고리명 및 설명 수정.
- **조치 완료**: `ko.json` 및 `en.json`을 수정하여 제목을 "급상승 쇼츠" (Trending Shorts), 부제를 "지금 빠르게 오르는 영상들" (Videos rising rapidly right now)로 변경.

### [완료] 13. 메인 페이지에 개인정보처리방침/이용약관 링크 없음
- **현상**: 라이브 사이트 확인 결과, 메인 랭킹 페이지(`/ko`)에 **footer 자체가 없음**
- 다운로드 페이지에는 `© 2026 Shorts100` 텍스트만 있고 **Privacy / Terms 링크 없음**
- **위험**: Google Play 정책상 앱 내에서 개인정보처리방침 접근이 반드시 가능해야 함 → 심사 리젝 사유
- **조치 완료**: 모든 페이지 하단에 개인정보처리방침/이용약관 링크가 포함된 공통 `Footer.tsx` 컴포넌트를 추가하고 `RankingList` 및 `download/page.tsx`에 연동 완료.

### [완료] 14. 쿠키 동의 배너 일부 텍스트 하드코딩 (다국어 미적용)
- **현상**: `CookieConsentBanner.tsx` 109행 `"닫기"`, 117행 `"선택 저장"` 등 한국어 하드코딩
- **조치 완료**: `messages/ko.json`, `messages/en.json`에 `close`, `settings`, `saveSelection` 번역을 추가하고 `t()` 함수를 통해 다국어 처리 완료.

### [완료] 15. AdsManager가 쿠키 동의 완료 여부를 확인하지 않음 (Update.md #9 보완)
- **현상**: 쿠키 동의 전에 광고가 이미 로드되어 **GDPR/CCPA 위반 가능성**
- **조치 완료**: `AdsManager.tsx`에서 `localStorage` 동의 값을 확인하고, 커스텀 이벤트(`shorts100_consent_updated`) 리스너를 연동하여 사용자가 명시적으로 광고 쿠키를 동의(`advertising: true`)한 경우에만 로드되도록 제어 로직 구현.

### [완료] 16. CORS origin에 `shorts100.com` 프로덕션 도메인 누락
- **현상**: `backend/app/main.py`에 정식 도메인 `https://shorts100.com`이 누락되어 있음
- **조치 완료**: `allow_origins`에 `"https://shorts100.com"` 및 `"https://www.shorts100.com"`을 추가하여 CORS 에러를 미연에 방지.

- **할 일**: 첫 진입 시 "← → 스와이프로 다음 영상" 힌트 1회 표시

### 6. 다운로드 이력 만료 토큰 처리

### 17. 영상 상세 페이지 다국어 미적용

- 영어 선택 시에도 `"인스타그램 다운로드 안내:"`, `"틱톡 서버 보안 정책으로 인해..."` 한국어로 노출
- **할 일**: `translations` 객체에 해당 텍스트 추가 및 `uiLang` 반영

### 19. Rising 페이지 디자인 불일치 (메인 페이지와 다른 레이아웃)
- **현상**: Rising 페이지(`/rising`)는 독립적인 `<header>` + 단순 흰색 배경 사용
- 메인 랭킹 페이지와 완전히 다른 스타일 → 같은 앱인지 인식하기 어려움
- **할 일**: `RankingList` 컴포넌트의 헤더를 공유하거나, Rising 페이지도 동일한 디자인 시스템 적용

### 20. 다운로드 이력 "열기" 동작 개선 (만료 토큰 무시)
- **현상**: `history/page.tsx` 173행에서 `log.original_url`로 연결 → 이것은 유튜브 원본 URL
- `file_token`으로 서버의 다운로드 파일에 접근하는 "열기" 버튼이 없음
- 만료 여부(`isExpired`) 표시는 하지만 만료된 항목의 버튼이 비활성화되지 않음 (Update.md #6 관련)
- **할 일**: 만료되지 않은 항목은 `file_token` 기반 "재다운로드" 버튼 제공, 만료 항목은 버튼 비활성화

### 21. APK 파일(7.3MB)이 git에 여전히 포함됨 (Update.md #7 보완)

- **할 일**: `.gitignore`에 `*.apk` 추가 + `git rm --cached frontend/public/shorts100.apk` 실행

## 🔧 백엔드 수집 로직 분석

### 26. "뉴" 버튼이 진짜 신규 영상만 보여주지는 않음

**수집 방식 확인 결과:**

```
collect_realtime_shorts (짝수 시간: 00:00, 02:00, ...)
  → YouTube API: order="viewCount", publishedAfter=now-6h  (인덱싱 지연 보정용)
  → Python 재필터: published_at >= now-2h 인 것만 남김
  → _upsert_video_batch: DB에 있으면 stats 업데이트, 없으면 INSERT
```

**핵심 동작:**
- DB에 이미 있는 영상도 다시 수집 → stats(조회수/좋아요)만 업데이트됨
- 2시간 내 발행된 영상이 **연속 cycle에 중복 등장**할 수 있음 (00:00에 발행 → 00:00 수집 + 02:00 수집 둘 다 포함)
- `_filter_existing_video_ids`(기존 영상 제외 함수) 존재하지만, 실시간/일간 수집에는 **적용 안 됨**

**화면 "실시간" 탭의 실제 동작:**
- API: `/api/rankings/global?period=realtime`
- 로직: Video 테이블에서 **지금 기준 24시간 이내 발행된 영상** 조회 (슬라이딩 윈도우)
- "실시간"이라는 이름이지만 24시간 전 영상까지 포함됨

**개선 방향:**
- `period=realtime` 레이블을 **"오늘(24h)"** 으로 변경하면 기대치와 실제 동작이 일치
- 진짜 "2시간 신규" 를 보여주려면 `/api/chart?chart_type=real` 엔드포인트(이미 구현됨)를 프론트에 연결해야 함
- **할 일 (단기)**: `RankingList.tsx` `periods.realtime` 레이블 → `"오늘"` / `"Today"` 로 변경
- **할 일 (장기)**: "2시간 NEW" 탭 추가 및 `/api/chart?chart_type=real` 연결
